# 知识领域整理机制

> L3 资源。PM 在 Step 7 复盘时检查扩展表是否需要整理。
> 扩展表见 `resources/research/taxonomy-extended.md`，正式表见 `resources/research/taxonomy.md`。

## 触发条件（满足任一）

- 扩展表条目 ≥5 条（未命中 + 候选合计）
- Step 7 复盘中 Leader 要求

## 整理步骤（PM 执行）

**1. 晋升**：扫描未命中 + 候选表 → 同一组合出现 ≥2 次且搜索结果有效 → 加入 taxonomy.md 映射表。记录到 taxonomy-extended.md §晋升记录。

**2. 合并**：搜索关键词或关注面高度重叠的条目 → 合并为一条，保留搜索效果更好的。判定标准：关键词相似度 >50%（如 "支付幂等" 和 "幂等键" → 合并）。

**3. 归档**：状态为"无效"且超过 6 个月的条目 → 移入 taxonomy-extended.md §归档。标注清理日期和原因。归档条目不再参与后续整理。

**4. 分类变更**：新业务领域/项目类型出现 ≥3 次 → PM 提议正式加入 taxonomy.md 分类体系，Leader 确认。

## 整理输出

PM 在扩展表末尾追加一条整理摘要：

```markdown
### [日期] 整理
- 触发: [条目超5 / Leader要求]
- 晋升: N 条 → taxonomy.md
- 合并: N 组 → N 条
- 归档: N 条
- 分类变更: [如有]
- 扩展表剩余: N 条
```
